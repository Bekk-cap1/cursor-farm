## 1. Цель документа

Зафиксировать **что должно быть реализовано**, чтобы платформа перестала быть «MVP с синтетикой» и стала **полноценным продуктом** для реальной эксплуатации: воспроизводимые данные, измеримая ценность для агронома/фермера, безопасность, сопровождение и масштабирование.

## 2. Определение «полноценного продукта» (Definition of Done)

Продукт считается **полноценным для первого коммерческого контура**, если одновременно выполнено:

1. **Данные доверяемы:** история показателей по полям и погоде хранится в БД, воспроизводима, с понятным происхождением (датчик / спутник / ручной ввод).
2. **Спутник и/или агрономия:** есть воспроизводимый конвейер **NDVI (или эквивалент)** по полигону поля с датами снимков и метаданными (облачность, источник); пользователь видит это на карте или в отчёте, а не только синтетический прокси.
3. **ИИ прозрачен:** ответы агента опираются на факты из БД/интеграций; при использовании LLM — политики стоимости, лимиты, логирование запросов, отказ при отсутствии данных.
4. **Безопасность:** роли соблюдаются на всех эндпоинтах; секреты в окружении; готовность к аудиту (журнал действий покрывает критичные операции).
5. **Эксплуатация:** развёртывание документировано (Docker или аналог), резервное копирование БД, миграции схемы, мониторинг ошибок и здоровья API.
6. **Качество:** автоматические тесты на критичные сценарии (авторизация, границы фермы, создание задач); CI на PR.

## 3. Продуктовые модули и требования по зрелости

### 3.1. Учёт и операции (уже в MVP — довести до «прод»)

| Модуль | Текущее состояние | Требование «полноценного продукта» |
|--------|-------------------|-----------------------------------|
| Фермы, участники, роли | Есть | Покрытие тестами; UI скрывает недоступные действия; единая матрица «роль × операция». |
| Поля (зоны), стада, задачи | Есть | Массовые операции (по желанию); экспорт списков (CSV); дедлайны и часовые пояса согласованы. |
| Уведомления | Есть | Настройка каналов (in-app минимум; email — фаза 2); не дублировать бессмысленно. |
| Аудит | Есть | Расширить события: вход, смена роли, удаление фермы, экспорт данных. |

### 3.2. Мониторинг полей и почвы

| Тема | Требование |
|------|------------|
| Текущие показания | Источник явно помечен: ручной ввод, импорт, интеграция датчика, расчёт по спутнику. |
| **История** | Таблица(ы) временных рядов: `zone_id`, `metric`, `value`, `recorded_at`, `source`, опционально `quality`. Не менее: влажность, при необходимости pH/EC/температура. |
| Синхронизация | API и UI не перезаписывают историю без версии/append-only для телеметрии. |
| Аналитика | Тренды и «стресс» считаются по **реальным точкам**, а не только по одному текущему значению (синтетика — опциональный fallback с дисклеймером). |

### 3.3. Спутник (NDVI / вегетация)

| Тема | Требование |
|------|------------|
| Входные данные | Полигон поля в WGS84 (GeoJSON или эквивалент), валидация площади и самопересечений. |
| Конвейер | Выбор источника: минимум **Copernicus Sentinel-2** (или коммерческий API с фиксацией в договоре). |
| Выход | Ряд NDVI (mean/min/max/std) по **снимку и дате**; статус (успех / облако / нет данных); ссылка на превью (PNG/tile) или bbox. |
| UX | Карта/слой NDVI на странице поля; легенда; переключение дат; дисклеймер про облака и задержки. |
| Ограничения | Квоты API, кэширование тайлов, фоновые задачи (очередь) при длинных запросах. |

### 3.4. Погода

| Тема | Требование |
|------|------------|
| Текущее | Open-Meteo или аналог зафиксировать в конфиге. |
| Продукт | Прогноз на N дней для координат фермы; при необходимости — история осадков для корреляции с поливом; единицы измерения и локаль. |

### 3.5. ИИ-агент и аналитика

| Тема | Требование |
|------|------------|
| Rule-based слой | Сохранить для предсказуемости и офлайн-сценариев. |
| LLM (опционально) | Конфигурируемый провайдер; промпт с инъекцией **только разрешённого контекста**; лимиты токенов; отсутствие утечки чужих ферм в промпт. |
| Объяснимость | Краткие «факты, на которых основан ответ» (список метрик/дат). |
| Страница AI аналитики | Согласовать с дашбордом: без дублирования без смысла; либо «сводка» vs «глубокий отчёт». |

## 4. Нефункциональные требования

### 4.1. Производительность и масштаб

- Время ответа API для типовых CRUD: целевой p95 < 300 ms при умеренной нагрузке (уточнить под инфраструктуру).
- Долгие задачи (спутник, отчёты): асинхронно (job id + polling или WebSocket).

### 4.2. Надёжность

- Резервное копирование SQLite/PostgreSQL по расписанию; процедура восстановления описана в README.
- Версионирование схемы БД (Alembic или эквивалент при переходе на Postgres).

### 4.3. Безопасность

- HTTPS в продакшене; заголовки безопасности (базовый набор).
- CORS: whitelist доменов продакшена, не `*` с куками.
- Хранение паролей: только хэш (bcrypt/argon2 — как принято в проекте).
- Rate limiting на login и дорогие эндпоинты (частично уже есть — расширить покрытие).

### 4.4. Наблюдаемость

- Структурированные логи (request id, user id, farm_id где уместно).
- Опционально: Sentry/OpenTelemetry для фронта и бэка.

### 4.5. Локализация и доступность

- i18n: полнота ключей для новых экранов (RU/EN минимум).
- Базовая a11y: фокус, контраст кнопок, подписи к полям форм.

## 5. Данные: целевая модель (дополнения к текущей)

Минимальный набор **новых сущностей** (концептуально):

1. **`telemetry_reading`** (или аналог): привязка к `FieldZone`, метрика, значение, время, источник.
2. **`ndvi_record`** (или аналог): привязка к полю, дата снимка, mean/min/max/std, источник, статус, ссылка на артефакт (опционально).
3. **`background_job`**: тип (спутник, отчёт), статус, ошибка, связь с сущностью.

Детальная схема — отдельный подпроект миграций после утверждения ТЗ.

## 7. Тестирование и CI

- **Backend:** pytest — авторизация, изоляция ферм, CRUD с ролями, расчёт аналитики на фикстурах.
- **Frontend:** минимум e2e (Playwright) — логин, открытие фермы, создание задачи; или критичные unit/integration.
- **CI:** на каждый PR — линт + тесты + сборка фронта.

## 9. Дорожная карта (фазы)

### Фаза A — «Доверие к данным» (без спутника)

- Персистентная история телеметрии по зонам; графики по истории.
- Уточнение UX пустых состояний и выбора фермы на дашборде (не только первая ферма).
- Тесты + CI; бэкапы.

### Фаза B — «Спутник NDVI»

- Полигон поля, конвейер Sentinel-2, хранение `ndvi_record`, слой на карте/превью.
- Отказ от чисто синтетического ряда как единственного источника для принятия решений (синтетика — демо-режим).

### Фаза C — «ИИ и продукт»

- LLM с контролем контекста и лимитами; улучшение объяснимости.
- Email-уведомления и сброс пароля (по бизнес-решению).
- Роли организации / мультитенант (если нужно B2B).

## 10. Критерии приёмки (верхний уровень)

Продукт принимается как **полноценный v1.0 для коммерции**, если:

1. История телеметрии по полям работает end-to-end, данные не теряются при обновлении показаний.
2. NDVI по полю получен хотя бы из одного поддерживаемого спутникового контура и отображается пользователю с метаданными.
3. Все критичные API проверены тестами на изоляцию между пользователями/фермами.
4. Есть документированное развёртывание и процедура бэкапа.
5. Юридический дисклеймер и политика данных доступны пользователю (ссылка в футере / при регистрации).

---

## 11. Вне объёма (явно)

- Полная замена агронома/ветеринара (продукт — помощник).
- Гарантия урожайности или финансовых показателей.
- Офлайн-first мобильное приложение (можно запланировать отдельно).
