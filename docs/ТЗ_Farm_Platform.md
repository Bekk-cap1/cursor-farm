## 1. Назначение и цели

### 1.1. Назначение

Веб-приложение для учёта и операционного управления сельскохозяйственным хозяйством: несколько ферм у пользователя или организации, поля (зоны), стада, задачи, сводки по дашборду, погода, диалог с ИИ-агентом с привязкой к данным хозяйства, уведомления в интерфейсе, журнал действий (аудит), управление участниками фермы и ролями.

### 1.2. Цели продукта

- Свести в одном месте **обзор по фермам**, **просроченные и сегодняшние задачи**, **поля и стада**.
- Дать **ролевой доступ** (владелец / менеджер / наблюдатель) к данным фермы.
- Обеспечить **прозрачность изменений** через аудит.
- Дать **ориентир по погоде** и **советы через ИИ** с явным ограничением ответственности (не замена агронома/ветеринара).

### 1.3. Границы первой очереди (MVP+)

Реализовано или заложено в текущей кодовой базе: авторизация JWT, CRUD ферм (с координатами), зоны полей, группы стада, задачи, дашборд, уведомления, погода (Open-Meteo), агент (правила + опционально LLM), аудит, участники фермы, i18n RU/EN, превью 3D полей со спутниковой подложкой (тайлы Esri).

## 2. Термины

| Термин | Описание |
|--------|----------|
| Ферма | Логическая единица хозяйства: название, регион, часовой пояс, опционально широта/долгота. |
| Зона поля (FieldZone) | Участок с культурой, площадью, типом полива, оценкой влажности почвы. |
| Стадо (HerdGroup) | Группа животных: вид, численность, заметки по кормлению. |
| Задача | Название, тип (полив / кормление / прочее), срок, статус. |
| Уведомление | Запись в колокольчике UI (просрочки, синхронизация с бэкендом). |
| Участник фермы | Связь пользователь–ферма с ролью. |

## 3. Акторы и роли

### 3.1. Пользователь системы

- Регистрация и вход по email/паролю.
- Принадлежность к одной или нескольким фермам через **membership**.

### 3.2. Роли на ферме

| Роль | Назначение (логика продукта) |
|------|------------------------------|
| owner | Полный контроль, удаление фермы, управление участниками (в рамках реализованного API). |
| manager | Редактирование данных фермы, полей, стада, задач. |
| viewer | Только просмотр (ограничения должны соблюдаться и в API, и в UI). |

**Требование:** любое изменяющее действие проверяет роль через единый слой авторизации (deps / политики).

## 4. Функциональные требования

### 4.1. Аутентификация и профиль

- Регистрация, вход, выход.
- JWT в заголовке `Authorization: Bearer`.
- Отображение email пользователя в шапке.

**Будущее (рекомендуется в ТЗ как отдельная фаза):** сброс пароля, смена пароля, политика сложности пароля, опционально 2FA для владельцев организаций.

### 4.2. Фермы

- Создание фермы: название, регион; опционально широта и долгота (для привязки карт/подложек).
- Просмотр списка доступных ферм с метриками (алерты, задачи на сегодня) — дашборд/API.
- Редактирование и удаление — по правилам роли и владения.

### 4.3. Дашборд

- Сводка: количество ферм, просроченные задачи, задачи на сегодня, агрегаты по полям и стадам.
- Недавние задачи с переходом к ферме.
- Блок превью 3D: мини-сцена полей со **спутниковой подложкой** по координатам выбранной/первой фермы; обязательная **атрибуция** источника тайлов (Esri World Imagery и др.).

### 4.4. Страница фермы

- Вкладки: обзор, поля, стада, задачи, ИИ-агент.
- Обзор: блоки полив/стадо, погода (текстовая сводка).
- Поля: форма добавления, список с редактированием полей, удаление с подтверждением.
- Стада: аналогично.
- Задачи: создание, отметка выполнения, отображение сроков.

### 4.5. Погода

- Запрос сводки по координатам фермы (внешний сервис Open-Meteo или аналог).
- Понятный дисклеймер в подвале: ориентировочность, решения с специалистами.

### 4.6. ИИ-агент

- Чат в контексте фермы: ответы с учётом полей, стада, погоды (логика `agent_runner`).
- Генерация/предложение задач и уведомлений — в соответствии с реализованной бизнес-логикой.
- Явное разделение: **советы не заменяют** очный осмотр агронома/ветеринара; закупки на сторонних площадках — по решению пользователя.

**Будущее:** лимиты токенов, логирование запросов для поддержки, выбор модели, отключение LLM при сбое.

### 4.7. Уведомления

- Колокольчик в шапке: список уведомлений, отметка прочитанным, синхронизация с сервером.
- События обновления UI при изменении данных (custom events / refetch).

**Будущее:** email, push, webhooks, настройки подписок.

### 4.8. Аудит

- Запись значимых действий (создание/изменение/удаление сущностей, входы при необходимости).
- Просмотр журнала уполномоченными ролями (эндпоинты и UI по мере развития).

### 4.9. Участники фермы

- Приглашение/назначение ролей (в объёме реализованного API).
- Отображение «ваша роль» в UI.

### 4.10. Интернационализация

- Языки: русский и английский; переключатель в боковой панели.
- Даты/время в формате, зависящем от языка/контекста фермы.

### 4.11. Ограничение частоты запросов

- Rate limiting на критичных маршрутах (например, аутентификация) — предотвращение brute-force.

## 5. Нефункциональные требования

### 5.1. Производительность

- Время ответа API для типовых CRUD < 500 ms при нормальной нагрузке на целевом железе.
- Фронтенд: ленивая загрузка тяжёлых модулей (Three.js) где применимо.

### 5.2. Доступность

- Базовая семантика кнопок, подписи к иконкам где необходимо.
- Учёт `prefers-reduced-motion` для тяжёлой анимации/3D (по политике продукта).

### 5.3. Совместимость

- Актуальные версии Chrome / Edge / Firefox; адаптив под мобильную ширину (боковое меню, шапка).

### 5.4. Надёжность

- Health-check эндпоинт для балансировщика и мониторинга.
- Резервное копирование БД в эксплуатации (вне кода, процедура эксплуатации).

### 5.5. Безопасность

- HTTPS в продакшене.
- Хеширование паролей (не хранить в открытом виде).
- CORS настройка под домены продакшена.
- Валидация входных данных (Pydantic / схемы фронта).

## 6. Модель данных (логическая)

Сущности (как минимум): `User`, `Farm`, `FarmMembership`, `FieldZone`, `HerdGroup`, `Task`, `Notification`, `AuditLog`. Связи: ферма владеется пользователем-владельцем; зоны/стада/задачи привязаны к `farm_id`; участники через `FarmMembership`.

Детальная схема — в миграциях/моделях репозитория (`models.py`).

## 7. API (общие правила)

- Префикс `/api`, JSON, коды ошибок HTTP по смыслу.

- Версионирование API: при смене контракта — версия в пути или заголовке (на будущее).

## 10. Критерии приёмки (примерные)

- Пользователь может зарегистрироваться, войти, создать ферму, добавить поле, стадо и задачу; задача отображается на дашборде и на странице фермы.
- Пользователь с ролью viewer не может выполнять операции редактирования (проверка API).
- Уведомления отображаются и помечаются прочитанными без ошибок.
- Дашборд показывает согласованные числа с данными ферм.
- Погода и агент возвращают осмысленный ответ при валидных данных фермы.
